## Objective

Define the **behavioural, semantic, continuity, ownership, and governance contract** for Savings Release before any further UI, Plan integration, Reality bridge, or Reflection narration occurs.

**Core principle:** Savings Release preserves emotional safety, financial truth, continuity trust, and user autonomy. The month evolves. The user does not fail the plan.

This stage is **not** a release UI build, Plan rewrite, reserve optimisation system, affordability modifier, debt suppression system, investment unlock system, or behavioural scoring system.

## 1. Canonical meaning of “release”

### 1.1 What Beynd recognises when release happens

When a user confirms Savings Release, Beynd records a **user-initiated reserve utilisation event**:

> The user chose to recognise that part of a saved or invested balance was **used to absorb pressure this month**, and that the **confirmed asset balance** should reflect that use.

Release is **truth preservation**, not problem resolution.

| Layer | What changes | What does not change |
|-------|--------------|----------------------|
| **Asset truth** | `goal.baseSaved` or `investment.baseBalance` (then recomputed `saved`/`balance`) | Income, expenses, payment schedule |
| **Cash-flow truth** | Nothing | `calcMonthlyLeftover`, overbudget arithmetic |
| **Plan truth** | Nothing directly | Step sequence, amounts, debt order |
| **Continuity record** | `S.savingsReleases[]`, optional `activityLog` row | User month context, Reality posture inputs |

### 1.2 Meaning comparison

| Meaning | Status | Notes |
|---------|--------|-------|
| **Reserve utilisation** | **Canonical** | Primary definition — user chose to use saved assets |
| **Temporary pressure absorption** | **Allowed** | Describes *why* user may release; not an outcome claim |
| **Continuity support** | **Allowed** | Month evolved; plan not failed |
| **Intentional tradeoff** | **Allowed** | Especially for goal/investment sources |
| **Emotional flexibility** | **Allowed (framing only)** | User agency, not system permission to spend |
| **Controlled destabilisation** | **Internal only** | Engineering term — never user-facing |
| **Adaptive survival** | **Forbidden (user copy)** | Implies crisis / failure framing |
| **Affordability repair** | **Forbidden** | Release does not repair affordability |
| **Optimisation** | **Forbidden** | Not “smart rebalancing” |

### 1.3 Explicitly rejected user-facing meanings

These must **never** appear in release copy, previews, toasts, Reflection, Home, or Ask:

| Rejected phrase / implication | Why |
|-------------------------------|-----|
| “Problem solved” | Overbudget and cash-flow truth remain |
| “Pressure removed” | Pressure may remain in Monthly Left |
| “Safe now” | Implies false relief |
| “Beynd fixed affordability” | Affordability is not mutated |
| “Back in balance” (from release alone) | Requires income/expense/payment change |
| “You can spend freely” | Encourages impulsive spending |
| “Emergency solved” | Buffer may be depleted truthfully, not magically restored |

## 2. Reserve hierarchy

### 2.1 Core rule

> **Liquid ≠ emotionally available.**  
> **Balance > 0 ≠ release-eligible by default.**

Eligibility is a **governance decision** expressed through reserve tier, user confirmation, and copy — not through liquidity inference alone.

### 2.2 Reserve hierarchy matrix

| Reserve category | Beynd representation today | Emotional availability | Liquidity (real world) | Continuity meaning | Release appropriateness | Default eligibility | Default suppression | Future user override |
|------------------|---------------------------|------------------------|------------------------|--------------------|-------------------------|----------------------|----------------------|---------------------|
| **Emergency buffer** | Emergency goal (`cat:'emergency'`, name heuristics) | High for *named* pressure absorption | Usually liquid | Safety layer doing its job | **Appropriate** with buffer copy | **Eligible** if saved > 0 | None when user confirms | Re-label goal (user edit) |
| **Flexible savings** | Goal (general) or `investment.type:'savings'` / short horizon | Medium — may feel “available” | Often liquid | Flexible reserve | **Cautious** | Eligible with tradeoff copy | Deprioritise in default sort | Optional `reserveRole` (future, audited) |
| **Sinking funds** | Named goals (holiday, car, etc.) | Low — earmarked | Often liquid | Planned future spend | **Cautious** | Eligible with goal tradeoff copy | Never default source | User must select explicitly |
| **Goal savings** | Any non-emergency goal with `saved > 0` | Low–medium | Varies | Progress toward intent | **Cautious** | Eligible with tradeoff copy | Not auto-suggested | Explicit source pick |
| **Cash ISA** | `investment.type:'isa'` (or `'stocks_isa'`) | Medium — user may treat as cash | Liquid with rules | Tax-wrapped reserve | **Cautious** | Eligible tier-2 | Not default over buffer/goals | Classify as flexible reserve (future) |
| **General investments** | Unlinked `investments[]` | Low for month-pressure use | Varies | Long-term growth | **Last-resort** | Eligible only with warning (future gate) | **Suppress** from default list | Explicit pick + confirm |
| **Linked investment goal** | Investment row with `goalId`; goal row blocked | Low — dual identity | Varies | Goal progress via investment | **Last-resort** | Investment row eligible | Extra linked copy required | Same |
| **Pension** | `investment.type:'pension'` | Very low | Illiquid / restricted | Retirement | **Forbidden default** | **Not eligible** until contract amendment | Hard suppress | Strong override + warning (future) |
| **Property reserve** | `investment.type:'property'` | Very low | Illiquid | Long-term asset | **Forbidden default** | **Not eligible** until contract amendment | Hard suppress | Same |
| **Locked savings** | Not modeled | N/A | Not liquid | Protected | **Forbidden** | Not eligible | N/A | User removes lock (future) |
| **Manually protected reserves** | Not modeled (could be user note / future flag) | User-declared low | Unknown | User intent | **Forbidden until declared** | Suppressed | Honoured | User unprotects |

### 2.3 Current code vs contract gap

**Today (81D-3):** eligibility = any goal (non-linked) or investment with balance > 0.  
**Contract:** tiered eligibility, sort order, and copy must align with table above before broad surfacing.

### 2.4 Default source ordering (contract target)

When multiple sources exist, default UI ordering must be:

1. Emergency buffer (if eligible)  
2. Flexible savings / short-term reserves  
3. General goal savings (explicit tradeoff)  
4. Cash ISA / savings-type investments (cautious)  
5. Other investments (last-resort, warning)  
6. Pension / property — **excluded** until explicit product decision  

Never default to first array index without tier sort.

## 3. Continuity law for release

### 3.1 What must remain true after release

| Truth domain | Post-release law |
|--------------|------------------|
| **Monthly Left** | **Unchanged** — release is not cash-flow repair |
| **Overbudget state** | **Unchanged** until income/expenses/payments change |
| **Plan sequencing** | **Unchanged** — no re-run mutation of `getMonthPlan` on apply |
| **Debt prioritisation** | **Unchanged** |
| **Affordability / suggestableRoom** | **Unchanged** |
| **breathingRoomFloor** | **Unchanged** |
| **Buffer truth** | **Must reflect depletion** — saved ↓; readiness may leave `BUFFER_COMPLETE` |
| **Net worth** | **Must reflect depletion** — honest asset reduction |
| **Reality posture inputs** | **Unchanged** — release is not a Reality save event |
| **Reflection / Stability semantics** | **No automatic change** — optional calm interpretation later |

### 3.2 Hard rules (continuity law)

1. **Release must not silently resolve overbudget.**  
2. **Release must not silently improve affordability.**  
3. **Release must not fake emergency readiness** (buffer display must show lower saved, not hide depletion).  
4. **Release must remain explainable** — user chose source, amount, reason; record persisted.  
5. **Release must remain reversible in meaning** — user may rebuild buffer/goals later; copy must not say “too late.”  
6. **Release must preserve user agency** — explicit confirm; no auto-apply; no required release to proceed.  
7. **Release must not resequence Plan** without ordinary re-render from unchanged inputs.  
8. **Release must not write interpretation to `S`** beyond financial event record (`savingsReleases[]`).  

### 3.3 “Reversible in meaning”

Undo of balances is **not required** for contract compliance. Continuity requires:

- No shame for releasing  
- No permanence language  
- Rebuild path remains visible (buffer step, goal progress)  
- Future optional undo stage must be audited separately

## 4. Ownership boundaries

### 4.1 Ownership map

```
┌─────────────────────────────────────────────────────────────────┐
│ Tier 0 — Financial truth (immutable by release apply)           │
│  Plan math · affordability · calcMonthlyLeftover · payments   │
└─────────────────────────────────────────────────────────────────┘
                              ▲
                              │ read-only observation
┌─────────────────────────────────────────────────────────────────┐
│ Savings Release — reserve flexibility EVENT (Tier 0 asset only) │
│  User confirm · baseSaved/baseBalance · savingsReleases[]       │
└─────────────────────────────────────────────────────────────────┘
         ▲                              │
         │ calm link (future)           │ read-only history
┌────────┴────────┐          ┌─────────┴──────────┐
│ Reality           │          │ Reflection /       │
│ difficult-month   │          │ Stability Room     │
│ context capture   │          │ long-term meaning  │
│ (no apply)        │          │ (opt-in, later)    │
└───────────────────┘          └────────────────────┘
         ▲
         │ orientation only
┌────────┴────────┐
│ Home            │
│ overbudget card │
│ → Plan (no CTA) │
└─────────────────┘
```

### 4.2 System responsibility table

| System | Owns | Must never own | May observe release | May narrate release |
|--------|------|----------------|---------------------|---------------------|
| **Plan** | Deterministic sequencing; overbudget truth; explicit release entry (when shown) | Emotional diagnosis; affordability mutation from release | Eligible sources via existing helpers | Tier 1 — factual preview only (“saved balance will reduce”) |
| **Reality** | Difficult-month context; protect/worried/success; below-plan sense-check | Apply release; suggest amounts; auto-open modal | Overbudget / pressure flags | Tier 1 — optional calm bridge to Plan (future, copy-only) |
| **Savings Release** | User-confirmed event; asset balance mutation; `savingsReleases[]` | Plan math; Monthly Left; orchestration | Self | Tier 1 — confirmation sheet copy |
| **Reflection** | Opt-in month understanding | Release apply; urgency; “you should release” | Release history (read-only, future) | Tier 3 — calm retrospective only, if useful |
| **Stability Room** | Calm explainability (future route) | Intervention; amounts | `noticed.*`, release history | Tier 3 — observational, non-prescriptive |
| **noticed.*** | Observational export vocabulary | User-facing release CTAs | Release counts (future, internal) | **No** direct narration |
| **Home** | Orientation; route to Plan when overbudget | Release modal; release prompts | Overbudget visibility | Tier 1 — “Review in Plan” only |

### 4.3 Anti-patterns (forbidden)

- Reality footer + Plan block + Home card all explaining release in the same session  
- Reflection auto-interpreting release on the same day as apply  
- Ask Beynd recommending release amounts without user request  
- `noticed.*` driving release eligibility  
- Orchestration promoting release to main action

## 5. Emotional framing

### 5.1 Framing table

| Phrase | Status | Use when |
|--------|--------|----------|
| “Record using part of your buffer” | **Approved** | Emergency buffer source |
| “Leaning on savings temporarily” | **Approved** | General release confirm |
| “Your saved balance will reduce” | **Approved** | Preview truth block (required) |
| “This does not change your income or spending lines” | **Approved** | Preview truth block (required) |
| “Savings are sometimes there to be used” | **Approved (cautious)** | Intro — not standalone relief |
| “Shifts progress on [goal name]” | **Approved** | Goal source |
| “Long-term tradeoff” | **Approved** | Investment source |
| “Holding flexibility this month” | **Discouraged** | Implies Monthly Left improved |
| “Creating breathing room” | **Discouraged** | Conflicts with non-mutation of breathingRoomFloor / Monthly Left |
| “Using saved money” | **Discouraged** | Too casual; sounds like spending permission |
| “Pressure covered” | **Discouraged** | Implies cash-flow fix — replace with “Amount recorded toward this month’s gap” if needed |
| “Fixes overbudget” | **Forbidden** | False |
| “Safe now” / “You’re back on track” | **Forbidden** | False without expense/income change |
| “Beynd noticed you need to release” | **Forbidden** | Surveillance + coercion |
| “Smart move” / “Optimised” | **Forbidden** | Optimisation framing |
| Therapy / failure / panic language | **Forbidden** | Emotional safety |

### 5.2 Tone rules

- **Observational**, not congratulatory  
- **Factual**, not reassuring-by-assertion  
- **Optional**, not corrective  
- **Tradeoff-visible** for goals and investments  
- **No shame** for tight months

## 6. Future release experience direction (non-build)

### 6.1 Allowed future surfaces (priority order)

| Surface | Role | Auto? |
|---------|------|-------|
| **Manual user action (Plan)** | Primary apply path | No — user opens + confirms |
| **Quiet contextual option (Plan overbudget)** | Secondary CTA | No |
| **Reality calm link** | “If savings absorbed pressure, record in Plan” | No — link only |
| **Release history (read-only)** | Audit trail for user trust | No |
| **Reflection retrospective** | Month-later calm meaning | No — opt-in surface |
| **Ask Beynd discussion** | Explain past release if user asks | No auto-send |
| **Stability Room** | Explain reserve role over time | No intervention |

### 6.2 Must never happen

- Aggressive Home / push prompts to release  
- Predictive release suggestions (“You should release £X”)  
- Automated reserve usage on overbudget detection  
- Hidden reserve optimisation behind scenes  
- Pressure-triggered coercion or blocking (“Release to continue”)  
- Release as default fix before editing spending  
- Tier-C or Reflection blocks triggered by release event

- Append `S.savingsReleases[]`  
- Append `activityLog` (`savings_release`)

## 9. Final recommendation

### 9.1 Is current release architecture philosophically safe?

**Partially yes, operationally incomplete.**

| Safe | Unsafe / incomplete |
|------|---------------------|
| Monthly Left not mutated | Flat eligibility treats pension = buffer |
| Plan math survives apply | “Use saved money” / “Pressure covered” copy |
| User confirmation required | No tier sort; first row default |
| Buffer truth depletes honestly | No release history UI |
| Persisted audit trail (`savingsReleases[]`) | Reality / Reflection unbounded if added carelessly |

**Verdict:** Mechanics align with **balance-truth mutation** contract. **Semantics and surfacing** do not yet align with this continuity contract.

### 9.2 Biggest unresolved risk

**Semantic flattening + false relief copy** — users believe release fixed the month’s cash-flow picture while Plan remains overbudget and Monthly Left unchanged. Secondary risk: **long-term investments presented as equally available as buffer**.

### 9.3 What must happen before further implementation

1. **80R-A.3 or 80R-D** — Source classification model mapped to existing fields (`cat`, `type`, `purpose`, `horizon`)  
2. **Copy pass contract** — Replace discouraged phrases; add mandatory truth block  
3. **Eligibility tier gate** — Suppress pension/property; deprioritise investments  
4. **End-to-end QA matrix** — Scenarios A–E from 80R-A.2  
5. **Release history read-only UI spec** — Before Reflection narration  

### 9.4 What should remain permanently shadow / internal

- Eligibility tier scoring  
- Cross-source optimisation  
- Predictive release hints  
- Orchestration-driven release promotion  
- `noticed.*` → release CTA wiring  
- Affordability repair shadow experiments  

### 9.5 Safest future implementation sequence

| Stage | Deliverable |
|-------|-------------|
| **80R-A.1** | This contract ✅ |
| **80R-B** | Source tier helpers + sort (read-only eligibility extension) |
| **80R-C** | Copy pass + truth block on existing modal (no new surfaces) |
| **80R-D** | Release history read-only UI |
| **80R-E** | Reality → Plan calm link (copy-only, no apply) |
| **80R-F** | Reflection retrospective spec + gate (optional, later) |
| **80R-G** | Ask Beynd export rules for release truth |

**Do not parallelise** Reflection narration with eligibility tier work.
