## Objective

Define the **safest future change** to fix Reflection entry completeness after 80N-Q confirmed that Reflection is **not code-gated on Protect**, but the UX behaves **Protect-weighted** because of collapsed summary, entry lifecycle, and field hierarchy.

This stage defines **product continuity law only**. It does **not** implement helpers, UI changes, or runtime wiring.

**Core principle:** Any user-authored month context field may qualify Reflection — but **depth, tone, and affordance** must match context strength. Silence remains the default when nothing meaningful exists.

## 1. Problem statement

### 1.1 Actual gate logic vs perceived UX

| Layer | Reality (code) | Perception (UX) |
|-------|----------------|-----------------|
| **Eligibility** | `hasAny = protect \|\| worried \|\| success` — equal OR | “Reflection needs Protect” |
| **Entry gate** | `geodeStabilityRoomEntryGate` — `monthContextHasAny` satisfies content threshold | Entry vanishes when Protect removed |
| **Experience compose** | `what_matters_this_month` builds lines for all filled fields | Collapsed card shows Protect only |
| **Inline affordance** | `suggested` only when `justSavedMonthContext` session flag set | One-shot; Edit clears flag |

**The gap is product continuity, not Plan math.** Gate logic already treats Worried and Success as meaningful. Presentation and lifecycle make Protect appear to be the hidden primary key.

### 1.2 Why Protect appears overweighted

1. **Collapsed summary** (`geodeRealityMonthContextCompactSummary`) — if `mc.protect` exists, show “In mind: …”; else generic “Month context saved” even when Worried/Success remain.
2. **Field hierarchy** — Protect is field #1, labelled “What matters most this month.”
3. **Carry summary** — prior-month line prioritises Protect before Worried/Success.
4. **Atmosphere bias** — Protect drives `protect_flexibility` posture → `protective` atmosphere; without Protect, tone shifts even when worry/hope remain.
5. **Ephemeral entry** — inline Reflection link tied to `justSavedMonthContext`; expanding the card clears it until re-save.

### 1.3 Why Worried and Success must count as meaningful month context

- Users often articulate **fear, uncertainty, or hope** before they can name what they are protecting.
- Month context is **user-authored coaching orientation**, not a financial defence declaration.
- Reflection’s role is to **hold context together calmly** — not to require a protective frame.
- Gate code already encodes this (`hasAny` OR). UX must catch up.

### 1.4 Why Reflection must not disappear when Protect is blank

Reflection should disappear only when:

- **No meaningful month context** exists (all three fields empty after trim), **and**
- No alternate content threshold is met (heldBack, plan focus, reality alignment — existing gate paths), **and**
- User has not dismissed the current suggestion (session/month rules below).

Removing Protect while Worried and/or Success remain is **not** a loss of meaningful context. The inline affordance must not imply otherwise.

**Exception (correct silence):** Protect was the **only** populated field and user clears it without replacing context — `hasAny` false → remain silent. This is correct, not a bug.

## 2. Context-completeness principle

### 2.1 Core rule

> **Reflection may be available when any meaningful month context exists:**
> - Protect  
> - Worried  
> - Success  

**Depth and tone vary by context strength.** Eligibility does not require Protect.

### 2.2 Strength model

| Context strength | Fields present | Reflection depth | Default silence |
|------------------|----------------|------------------|-----------------|
| **None** | All empty | Hidden | **Yes** — remain silent |
| **Light** | Exactly one field (worried-only, success-only, or protect-only) | Light reflection (1–2 blocks) | No — available when gate passes |
| **Standard** | Two or more month-context fields | Standard reflection (up to 3 blocks) | No |
| **Protective / Recovery** | Protect and/or heldBack and/or strong plan focus | Fuller reflection; protective/recovery atmosphere | No — still opt-in, never auto-open |

### 2.3 Interpretation rules

- **Single-field context = light reflection** — shorter, fewer blocks, no Tier-C, no background unless heldBack exists independently.
- **Multi-field context = fuller reflection** — existing max-3 block law applies.
- **Protect + heldBack + plan focus = stronger reflection** — atmosphere may be protective/recovering; Tier-C remains governed and rare.
- **No meaningful context = remain silent** — no inline nag, no auto-open, no Home/Plan prompts.

### 2.4 Non-negotiables

- User-authored words **echo only** in Reflection — never re-diagnose or escalate.
- Silence-first: absence of context is not a failure state.
- Emotional safety: worry is not instability; success is not shallow.

## 3. Collapsed summary rule (Reality card)

### 3.1 Current problem

Protect-only summary creates **Protect-as-anchor bias** in the collapsed “This month” card.

### 3.2 Future rule — first meaningful field

Show the **first non-empty field** in this priority order for **single-line summary**:

1. Protect  
2. Worried  
3. Success  

| Field | Summary prefix | Example |
|-------|----------------|---------|
| Protect | `In mind:` | `In mind: rent and family support` |
| Worried | `On your mind:` | `On your mind: overspending this month` |
| Success | `Success would feel like:` | `Success would feel like: staying steady` |

**If multiple fields exist:** prefer the **first in order above** for the collapsed one-liner. Optionally append a calm secondary hint (implementation detail, Phase C):

- `In mind: rent · also watching bills` — only when Protect + Worried both present and combined length ≤ ~72 chars after truncation.
- Do **not** show three-field walls of text in collapsed state.

### 3.3 Copy rules

- **Never** imply worry is failure, weakness, or instability.
- **Never** imply success is less important or “soft.”
- **Never** over-personalise (no “you seem anxious”, no therapy framing).
- **Preserve calm tone** — observational, not evaluative.
- **Truncate** long values consistently (existing ~52 char pattern acceptable).

### 3.4 Fallback when `hasAny` but summary logic fails

If `hasAny === true` but no field string survives trim (should not happen):

- Show: `Month context saved` (existing neutral fallback).

## 4. Reflection entry lifecycle

### 4.1 Current behaviour (baseline)

| Mechanism | Behaviour |
|-----------|-------------|
| `suggested` | Inline `›` link after **Save month context** when `justSavedMonthContext === true` |
| `available` | Gate state exists; **does not render** inline link today |
| Edit expand | Clears `justSavedMonthContext` |
| `Not now` | Session dismiss; cleared on next save |
| Reflection surface | Opens only via inline tap; no persistent manual tab |

### 4.2 Options assessed

| Option | Description | User friction | Nag risk | Silence-first | Implementation risk |
|--------|-------------|---------------|----------|---------------|----------------------|
| **A** | Keep only suggested one-shot | Low on nag | **Low nag** | **High compliance** | **Low** — status quo |
| **B** | Calm `available` affordance when `hasAny` && !dismissed | **Low** — user chooses | **Medium** if copy urgent | **Medium** — needs calm copy | **Medium** — gate + HTML |
| **C** | Entry inside collapsed card only | Low | Low–medium | **High** if card-scoped | **Low** — aligns with 80J-E |
| **D** | Depth-tier entry labels | Low | Low if labels calm | **High** if tier = tone not nag | **Medium** — label map |

### 4.3 Recommendation (hybrid — B + C + partial D)

**Adopt:**

1. **Keep `suggested` after save** — one calm inline link immediately post-save (existing 80H-B law).
2. **Add quiet `available` affordance** in collapsed month-context card when:
   - `hasAny === true`
   - Gate `visibility !== 'hidden'`
   - User has **not** dismissed for current session/month (see §7)
   - `justSavedMonthContext` is false (avoid duplicate links post-save)
3. **Card-scoped only** — no Home, no Plan, no global banner.
4. **Depth-tier labels** for affordance copy only (not urgency):

| Depth | Available label (example) | Suggested label (post-save, example) |
|-------|----------------------------|--------------------------------------|
| Light | `See a quieter read of this month ›` | `See how this month fits together ›` |
| Standard | `See how this month fits together ›` | Same |
| Protective/Recovery | `See how this month fits together ›` | Same |

**Do not:**

- Auto-open Reflection on save or on field change.
- Show available affordance when `hasAny === false`.
- Re-show dismissed suggestion until next save or meaningful context change (§7).

## 5. Reflection depth tiers

### 5.1 Tier Light

**When:** worried-only, success-only, or protect-only (exactly one populated field).

| Aspect | Rule |
|--------|------|
| **Entry label** | `See a quieter read of this month ›` (available); post-save may use standard suggest copy |
| **Max blocks** | 1–2 (prefer `what_matters` + one atmosphere block) |
| **Tier-C** | **Forbidden** — gate must not PASS for Tier-C on Light tier |
| **background** | Only if heldBack signals exist **independently** of month context |
| **Tone** | Calm, declarative, minimal |
| **Suppressions** | Cross-surface pacing suppress applies to `month_so_far` as today |

**Forbidden copy:** urgency, “you should”, diagnosis, protect framing when user only expressed worry/hope.

### 5.2 Tier Standard

**When:** two or more month-context fields; or one field + heldBack/reality/plan threshold without protective recovery signals.

| Aspect | Rule |
|--------|------|
| **Entry label** | `See how this month fits together ›` |
| **Max blocks** | 3 (existing law) |
| **Tier-C** | Governed — suppression-first; PASS remains rare |
| **background** | Per atmosphere (reflective/recovering pick) |
| **Tone** | Reflective, guidance register |
| **Suppressions** | Full 80N-N.1 cross-surface law |

### 5.3 Tier Protective / Recovery

**When:** Protect present **and** (heldBack signals **or** protective atmosphere **or** plan focus with alignment).

| Aspect | Rule |
|--------|------|
| **Entry label** | `See how this month fits together ›` — **not** “protect your finances” |
| **Max blocks** | 3 |
| **Tier-C** | Existing gate only — **no increase in visibility** |
| **background** | Preferred over `plan_protecting` when recovering + heldBack |
| **Tone** | Protective/recovery registers — calm, not alarm |
| **Suppressions** | Tier-C HARD_SUPPRESS vs background; cross-surface pacing active |

**Forbidden copy:** alarm, “at risk”, “you need to protect”, therapy language, second protective lecture.

## 6. Block eligibility by context

Legend: **Y** = may show (subject to gate + atmosphere + suppressions), **N** = should not show, **—** = not applicable.

| Context profile | what_matters | month_so_far | background | plan_protecting | noticed_tier_c |
|-----------------|-------------|--------------|------------|-----------------|----------------|
| protect-only | Y | Y | N* | Y | N† |
| worried-only | Y | Y (light tone) | N* | Y‡ | N† |
| success-only | Y | Y (light tone) | N* | Y‡ | N† |
| protect + worried | Y | Y | Y§ | Y | N† |
| worried + success | Y | Y | N* | Y‡ | N† |
| all three | Y | Y | Y§ | Y | N† |
| heldBack only (no month context) | N | Y | Y | Y | N† |
| no month context | N | Y¶ | Y§ | Y | N† |

\* background only if heldBack signals exist independently.  
† Tier-C: existing gate; PASS rare; **forbidden on Light tier**.  
‡ plan_protecting may show as orientation echo; lower priority in Light tier ordering.  
§ background when atmosphere recovering/reflective and heldBack present.  
¶ month_so_far atmosphere-only when other content thresholds met (existing gate).

### Block-specific rules (unchanged intent)

- **what_matters_this_month** — echo user words for **each filled field**; never Protect-only filter.
- **month_so_far** — atmosphere-driven; cross-surface neutralisation per 80N-N.1.
- **background** — heldBack only; never invent quiet.
- **plan_protecting** — plan orientation echo; not a Protect requirement.
- **noticed_tier_c** — replace `month_so_far` only on PASS; no expansion in this arc.

## 7. Not Now behaviour

### 7.1 Current intended behaviour (audit)

| Flag | Storage | Effect today |
|------|---------|--------------|
| `geodeReflectionEntryDismissed` | sessionStorage | Blocks `suggested` until cleared |
| Clear on save | `geodeClearReflectionEntryDismissed()` on month-context save | Re-enables suggest on next save |

**Edit expand** clears `justSavedMonthContext` but **does not** set dismiss.

### 7.2 Recommended contract (future)

| Event | Behaviour |
|-------|-----------|
| User taps **Not now** | Hide **suggested** inline link for **current session** |
| Same session, no save | **No** repeated suggest nag; **available** affordance may remain if spec Phase C adopts quiet manual access |
| User saves month context again | Clear dismiss; allow **one** new suggested link (existing law) |
| Meaningful context change | Clearing all fields then re-adding counts as new save event |
| Auto-open | **Never** |

**Recommended dismiss scope:**

- **Session dismiss** for `suggested` (keep current).
- **Do not** persist month-long dismiss for `available` — manual access stays calm and optional.
- If user dismisses and returns to collapsed card, show **available** label only (quieter copy), not the post-save suggest variant.

**No repeated nagging:** max one suggest per save event; available link is static text, not toast/banner.

## 8. Safeguards (explicit forbiddens)

The following are **out of scope** for 80N-Q.1 implementation and **must not** be introduced:

- Auto-opening Reflection on save, field blur, or month rollover
- Home entry prompts for Reflection
- Plan entry prompts for Reflection
- Emotional urgency (“important”, “don’t miss”, “act now”)
- Persistent “you should reflect” nudges or badges
- Requiring Protect for eligibility or copy
- Interpreting worry as instability or failure
- Treating success as shallow, secondary, or non-coaching
- Increasing Tier-C visibility or PASS rate
- Weakening cross-surface pacing suppression (80N-N.1)
- Therapy drift, diagnosis, or over-personalisation
- Persistence schema changes for dismiss flags (session-only unless future governance stage)
- Changes to Plan math, Home orchestration, Reality persistence, or financial calculations
